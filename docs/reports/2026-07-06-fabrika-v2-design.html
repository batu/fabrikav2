Build/CI fragmentation

Hard numbersDoc-cited

3 of 4 games import @fabrika/core without declaring it as a dependency — it resolves only via npm workspace hoisting, with no built/versioned core artifact (consumption is always live source; a breaking change in packages/core/src is invisible until each game's own tsc/vitest runs). The phantom deps, unbuilt/unversioned core, broken audit script, and toolchain drift remain true and unfixed in v1.

03

v2 architecture

Rendered from docs/architecture/v2-architecture.md as locked — this section presents it, it doesn't re-litigate it.

Thesis

v1's cost was not gameplay code — it was the commercial shell. 52% of all commits carry fix/rework language and they cluster on find_the_dog's in-canvas UI, while the shared package got 64× less churn and was mostly bypassed. v2 ships the shell — screens, SDKs, theming, build — as the product, and keeps games down to a canvas plus declarative config. Every design value lives in a sheet-driven generated layer, never in hand-written TS/CSS.

Guardrails — enforced, not aspirational

  1. Declared deps only. No phantom @fabrika/* imports; CI fails on undeclared. (Directly answers §2 item 4.)
  2. No literal design values in packages/ui or game shell code — lint greps for hex/rgba/copy-string patterns outside design/, replacing the broken grep-affected-games.sh with tests.
  3. Duplication gate. A game implementing something whose name/shape exists in packages/ fails tools/audit — "extract after 2nd use" inverts to "shell code starts shared."
  4. CI matrix covers every workspace from day one. (The architecture doc says "v1 covered 2 of 5" — true when written, but v1 commit b2a18cf39 has since added all five jobs; the guardrail stands as "never retrofit coverage," per §2 item 4.)
  5. Single toolchain versions pinned at root (v1 had vitest 3 vs 4, Capacitor drift).

05

design-sheets wiring

The reskin bar is zero code edits (locked decision). Here is the exact mechanism that makes that true, and what's still missing to get there.

The mechanism

  1. Standardize where design values live. v2 puts every game's design surface at games/<g>/design/ (tokens.css, copy.ts, assets.ts) plus --fab-* CSS tokens, so one generic ingester (fabrikav2 ingester) replaces v1's per-game bespoke ingester. It reads game.config.ts + design/ + ui-kit component cards.
  2. design-sheets gains new capabilities (schema changes explicitly allowed per the kickoff decisions): a copy schema (localized-copy leaf type — today nothing models UI text), asset-binding apply for module imports (v0 apply explicitly refuses these), and optional layout/spacing token conventions.
  3. Round-trip: dsheets build publishes the sheet → claude.ai/design (Batu, agents, or an external designer edit it) → dsheets diff/apply writes tokens/copy/asset bindings back into games/<g>/design/ atomically → CI builds + screenshot diff. Structural/component changes route to a change-brief card on the TWF board instead of silent auto-apply.
  4. New-game creation runs the same path in reverse. create-game scaffolds a default sheet, so picking a color scheme and assets for a new game is a sheet-editing session, not a code session — this is what puts theming-at-creation on the time-to-ship critical path.

v2-architecture.md:85–104
