## 2. Architecture as-built

- **Monorepo, npm workspaces.** Root `package.json:10-14`: workspaces = `packages/*`, `games/*`, `games/marble_run/*` (the last entry exists only because `marble_run` is a glob-matched variant folder, not a normal single-game workspace).
- **`packages/core`** (`@fabrika/core`, `packages/core/package.json`) is source-shipped (`"main": "src/index.ts"`, no build step), `private: true`, with 22 subpath `exports` covering `ui`, `runtime`, `debug`, `testing`, `playwright`, `ads`, `attribution`, `assets`, `haptics`, `shell`, and multiple `puzzle/*` subpaths. Directory listing (`packages/core/src`): `ads, assets, attribution, debug, haptics, index.ts, playwright, puzzle, runtime, shell, testing, ui`.
- **`games/`** currently holds exactly four active entries per `repo_guide.md:28-32`: `arrow` and `block_blast` are "ADOPTER" (active consumers of `@fabrika/core`); `find_the_dog` and `marble_run` are "FROZEN" ("Game status ... commit `017a87254`", `repo_guide.md:46-49`). `games/marble_run/` itself contains `sugar3d/` (the live variant) plus `archived_variants/` (dead, per directory listing `games/marble_run/archived_variants/2026-06-29`).
- **Build/run per game** (verified in each game's `package.json`):
  - `games/arrow/package.json`: `"dev": "vite --port 5181 --strictPort"`, `"build": "tsc --noEmit && vite build"`, `"test:unit": "vitest run"`, plus level-authoring scripts (`levels:gen`, `levels:check`, `level:new`).
  - `games/block_blast/package.json`: `"dev": "node ../../tools/claim-port.mjs 5173 && vite --port 5173 --strictPort"`, `"typecheck": "tsc --noEmit -p tsconfig.json && tsc --noEmit -p tsconfig.tests.json"`, `"test:e2e": "playwright test"`, `"test:unit": "vitest run"`, Android build via `"android:debug": "npm run build && npx cap sync android && npx cap run android"`.
  - `games/find_the_dog/package.json`: much larger script surface ‚Äî build chains through `tsc --noEmit && vite build` plus a sequence of `node tools/*.mjs` steps (`package-dist-level-assets`, `validate-dist-level-assets`, `ensure-android-portrait`, `apply-android-resources`, provider-validation/release-hook verifiers), separate `build:android`/`build:ios` variants, a `dashboard:*` script family for an `analytics-dashboard/`, remote-config generation/check scripts, level-schema codegen/validation scripts, and Android/iOS native-patch scripts (`android:sync`, `ios:sync`, `ios:next-build`). This is qualitatively a different (much heavier) build system than the other three games.
  - `games/marble_run/sugar3d/package.json`: Vite + `tsx` level generation (`gen:levels`), Playwright with a font-capture recording mode, `android:debug` chain identical in shape to block_blast's.
  - Root `package.json:5-9` only defines `typecheck`, `test:unit`, `lint`, each implemented as `npm run <script> --workspaces --if-present` ‚Äî i.e., a thin fan-out, added recently (file mtime Jul 2, matching the architecture review's Phase-0 recommendation date; see divergence note in ¬ß3).
- **CI** (`.github/workflows/ci.yml`): two jobs only ‚Äî `core` (typecheck + vitest for `@fabrika/core`) and `find_the_dog` (typecheck + lint). Comments in the file (lines 3-13) explicitly state Playwright e2e is intentionally excluded because "a first CI run proved the suite is not runner-portable," pointing to `todos/041` and `todos/040`. No CI job exists for `arrow`, `block_blast`, or `marble_run`.
- **Contracts:** `contracts/` contains exactly one file, `contracts/solver-metrics.schema.json` ‚Äî not a broad cross-game contract layer.
- **tools/** (repo-level): `claim-port.mjs`, `drive-device.js`, `drive-webview.mjs`, `play_ftd.js`, plus subdirectories `ftd-battery-ab/`, `git-lfs-worker/` (the Cloudflare R2 LFS backend, see `repo_guide.md:156-178`), `schema-migrations/`.
- **pw_sdk/**: `cropped_screenshots/`, `facebook/`, and a single markdown note `playwill-sdk-onboarding-phaser-notes.md` ‚Äî looks like an in-progress ad-network SDK onboarding scratch area, not a mature package.
- **Python backend:** lives entirely inside `games/find_the_dog/pipeline/` (see ¬ß5) ‚Äî there is no repo-level Python backend outside a single game.
- **Docs volume**: `docs/plans/` has ~140 dated plan files, `docs/brainstorms/` ~40, `docs/retros/` 14, `docs/solutions/` ~20 (plus subfolders), `docs/evidence/` ~35 evidence folders, and `features/` has 47 feature folders (`ls features | wc -l` ‚Üí 47) ‚Äî nearly all Find the Dog-specific by name (`find_the_dog_*`) or marble_run/sugar3d-specific. This is a heavily-FTD-weighted documentation corpus.
- `todos/` has 45 files (`ls todos | wc -l` ‚Üí 45), almost entirely Find the Dog levelbuilder/editor bugs (e.g. `018-pending-p1-apierror-toast-remount-wipes-visible-toasts.md`, `052-pending-p1-shop-restore-controls-throws-on-settings-page.md`).

## 3. Where docs' intent diverges from the code as-built

All divergences below are analyzed and cited firsthand in `docs/reports/2026-07-02-architecture-review-shared-core.md` (an internal audit, not my synthesis ‚Äî I am relaying its cited findings, cross-checked against the current tree):

- **"Shared code in `@fabrika/core`" is true for the puzzle engine, false for the commercial shell.** Audio is reimplemented 4√ó (~1,300 lines: `block_blast/src/audio/ProceduralSfx.ts` 337 + `SoundFx.ts` 44; `find_the_dog/src/audio/AudioManager.ts` 458 + `AmbientManager.ts`; `marble_run/sugar3d/src/audio/Music.ts` 145 + `Sfx.ts` 245; `arrow/src/game/audio.ts` 80) ‚Äî cited at report line 36. Analytics has zero shared code between block_blast's and FTD's near-identical type skeletons (line 37). Ads/attribution/IAP: core's `AdService` only covers AdMob; FTD's richer AppLovin Max/Adjust/RevenueCat stack has one consumer and lives outside core (line 38). Haptics: `@fabrika/core/haptics` exists yet 3 of 5 games reimplement it anyway (line 39).
- **"Two-game promotion rule" is itself violated by `puzzle/marble-board`** ‚Äî one consumer only, per report line 57 and 144; report recommends moving it out of core or writing an explicit grandfather justification.
- **Core has game-specific (FTD) leakage** despite being positioned as shared: hardcoded `ftd_manifest_cached`, `ftd_anon_id`, `ftd_cohort_` prefixes and `'ftd-assets'` IndexedDB name in `assets/manifest.ts:19-21`, `assets/cohort.ts:20-21`, `assets/cache.ts:181`; FTD-shaped UI timing defaults in `ui/index.ts:247,406,456,650` (report line 55-56).
- **"Phaser 3 + TypeScript" (repo_guide.md:5) is not what's actually shipped.** Only 3 of 5 games use Phaser; `arrow` is Canvas2D and `marble_run/sugar3d` is Three.js (report line 61, table at line 193-199 confirms substrate split). Nothing in the docs declares this as intentional policy vs. drift ‚Äî the report calls this out explicitly as "undeclared."
- **No scaffold/codegen despite a documented pipeline.** `repo_guide.md:86-94` describes a T1-T5 pipeline including a "phaser-core skill," but new games are created by manually copying a sibling per a markdown plan (report cites `docs/plans/2026-04-27-003-scaffold-arrow-colors-plan.md` at line 13/43).
- **No root typecheck/lint/test command ‚Äî partially fixed since the review.** The architecture review (dated 2026-07-02) states at line 13/100 "there is no shared tsconfig base, no root typecheck/test command." As of this research, root `package.json:5-9` *does* now define `typecheck`, `test:unit`, `lint` as workspace fan-outs ‚Äî file mtime (Jul 2 22:55) suggests this Phase-0 item was actioned after or during the review. However, the review's other Phase-0 items remain unaddressed as of this check: no root `tsconfig.base.json` (`find . -maxdepth 2 -iname tsconfig*` outside node_modules returns nothing), no root ESLint config, `packages/core/package.json.tmp` (0 bytes) is still present, and the dead `games/marble_run/archived_variants/` directory still exists on disk.
- **CI is far narrower than "typecheck before every commit" policy implies.** Report line 52: "block_blast, arrow_colors, marble_run have zero CI." Confirmed current `.github/workflows/ci.yml` only runs `core` and `find_the_dog` jobs.
- **Version drift contradicts an implied "one toolchain" monorepo norm.** Confirmed directly: `arrow`/`packages/core` pin `vitest ^3.2.0`, while `block_blast`/`marble_run/sugar3d` pin `vitest ^4.0.18`; Capacitor versions range `^8.1.0` vs `^8.3.1` across games' `package.json` files (matches report line 48).
- **Undeclared dependency on `@fabrika/core`.** Report line 50 states `find_the_dog`, `block_blast`, `arrow` import `@fabrika/core` without declaring it in `package.json` (works only via npm hoisting). Confirmed: `games/arrow/package.json` and `games/block_blast/package.json` list no `@fabrika/core` dependency at all, while `games/marble_run/sugar3d/package.json` does declare `"@fabrika/core": "*"`.
- **The repo-decomposition plan (2026-04-29) describes a game roster that no longer exists under those names.** `docs/brainstorms/2026-04-29-repo-decomposition-requirements.md:35-37` inventories nine games including `arrow_attack`, `pixel_flow`, `pixel_flow_gameplay`, `sandy_siphon`, `tilt_to_survive` ‚Äî none of these appear in the current `games/` directory (which holds only `arrow`, `block_blast`, `find_the_dog`, `marble_run`). The linked plan (`docs/plans/2026-04-29-001-refactor-repo-decomposition-plan.md`) has YAML frontmatter `status: draft-report` ‚Äî it was never marked executed, and the named games appear to have been abandoned/consolidated rather than extracted into standalone repos as the plan intended. This is a stale artifact: treat its topology claims as historical, not current.
- **Find the Dog's DOM/legal split is explicitly flagged by the docs themselves as a divergence risk**, not something I inferred: `repo_guide.md:154`: "Find the Dog store/legal pages are split across repos ... Do not claim public URLs are live until that repo is explicitly deployed" ‚Äî a self-aware warning against overclaiming cross-repo state.

## 4. Existing v2 / re-architecture / de-bloat notes already written

This is the most load-bearing finding for a v2 design ‚Äî substantial re-architecture work already exists in-repo:

- **`docs/reports/2026-07-02-architecture-review-shared-core.md`** (full text read) ‚Äî a two-part, evidence-cited internal audit:
  - Part 1 (lines 1-181): proposes a layered `@fabrika/core` (L0 kernel / L1 services / L2 Phaser engine / L3 testkit / L4 UI) plus a repo-level `platform/` layer (`templates/phaser-game/`, `tools/create-game.mjs`, a port registry, shared tsconfig/eslint/vite/playwright configs, workspace-generic CI). Five phases with effort estimates (Phase 0 ~1wk hygiene, Phase 1 ~3-4wk commercial-shell extraction, Phase 2 ~1-2wk scaffold, Phase 3 ~1wk CI matrix, Phase 4 ~2wk release-pipeline generalization, deferred until game #2 ships to a store).
  - Part 2 (lines 183-268), "why UI keeps getting reinvented": documents 3 incompatible meta-UI substrates across 5 games (table at line 193-199) and a duplication scoreboard (Button √ó5, modal √ó5, HUD √ó4, etc., line 203). Proposes declaring **DOM as the standard substrate for all meta-game UI** (line 215) and growing `core/ui` from 3 components to a ~14-component "game-shell UI kit" extracted from Find the Dog's existing DOM implementation, plus a `core/shell` flow-state-machine layer.
- **`docs/brainstorms/2026-04-29-repo-decomposition-requirements.md`** and its companion **`docs/plans/2026-04-29-001-refactor-repo-decomposition-plan.md`** ‚Äî a full "split every game into its own repo, keep this repo as shared studio platform" proposal (decisions D1-D6, requirements R1-R8, phased extraction order sandy_siphon ‚Üí arrow_attack/arrow_colors ‚Üí block_blast/tilt_to_survive/pixel_flow(_gameplay) ‚Üí arrow ‚Üí find_the_dog-last). - **`docs/brainstorms/2026-04-29-agentic-production-repo-approaches.md`** ‚Äî sibling brainstorm, not read in full here but co-located with the decomposition requirements; likely explores alternative repo-structuring approaches (worth a follow-up read if v2 planning needs the alternatives considered/rejected).

## 5. Python backend

- There is **no repo-level Python backend/service** for fabrika as a whole. The only Python code lives inside **`games/find_the_dog/pipeline/`** ‚Äî confirmed via `pyproject.toml` (`games/find_the_dog/pipeline/pyproject.toml`): package name `dog-pipeline`, `requires-python = ">=3.11"`, dependencies include `fastapi>=0.136.0`, `uvicorn>=0.44.0`, `sse-starlette` (SSE streaming), `Pillow`, `opencv-python-headless`, `rembg[cpu]` (background removal), with an optional `sam2` extra (`sam2`, `torch`, `torchvision` ‚Äî Meta's Segment Anything 2, for image-cutout generation).
  - This is the **level-generation/levelbuilder backend** for Find the Dog's AI-authored levels: an internal FastAPI dev server plus a React editor UI, not a production runtime backend for the shipped game. Files under `games/find_the_dog/pipeline/levelbuilder/api/` include `server.py`, `routes.py`, `job_store.py` (a local SQLite durable-job ledger, per `repo_guide.md:78`), `job_worker.py`, `inpaint.py`, `layer_provider.py`, `sequence_activation.py`, `remote_config_publisher.py`, `band_generation.py`, `canonical_migration.py`, `smart_hitboxes.py` ‚Äî matching the narrative in `repo_guide.md:76-83` about durable jobs, sequence activation/rollback, and Remote Config publishing.
  - Separately, `games/find_the_dog/analytics-worker/` is a **TypeScript** (not Python) Cloudflare Worker for analytics ingest/query (`src/ingest.ts`, `src/query.ts`, `src/storage.ts`, `src/budget.ts`, `src/contracts.ts`) ‚Äî the "owned analytics mirror" referenced by `docs/evidence/2026-07-02-ftd-owned-analytics-mirror`.
- **Sibling-repo dependency confirmed:** `games/find_the_dog/pipeline/pyproject.toml` lines 30-41 declare `merceka-core` as an editable path dependency in the `dev` dependency group: `merceka-core = { path = "../../../../merceka-core", editable = true }`, with a comment "BaSe layout: merceka-core sits next to fabrika/ (see ../../.. from this file)." This is dev/notebook tooling only (Jupyter, LLM wrappers), not a runtime dependency of the shipped game or the FastAPI server's `[project.dependencies]` list.
- I inspected `/Users/base/dev/appletolye/merceka-core/README.md` directly: it describes itself as "Core utilities for merceka projects" ‚Äî a generic, cross-project Python library (`merceka_core.llm.LLM` wrapper for Ollama/OpenRouter/Gemini, a cross-process `gpu_lock()` file-lock utility, an exception hierarchy for video/GPU errors) built with `nbdev`. It is **not fabrika-specific** and is not described anywhere in its own README as fabrika's backend ‚Äî it's a shared utility library consumed by multiple sibling projects at `/Users/base/dev/appletolye/` (its own text mentions "slab vision triage, mindweaver enrichment" as other consumers), of which fabrika/find_the_dog's pipeline is one dev-time consumer.
- A grep for "merceka" across fabrika's `*.py`/`*.md` (excluding the toml I found directly) returned only documentation mentions inside `docs/plans/`, `docs/brainstorms/`, and `docs/solutions/` ‚Äî none of which describe merceka-core as fabrika's backend; they're plan/brainstorm files about the FTD pipeline's own architecture (nbsync, notebook module sync, AI level generation) that happen to be filed under similarly-named directories. I did not find any file that identifies merceka-core as "the fabrika backend" ‚Äî my conclusion above (dev-tooling dependency, not runtime backend) is based on the direct `pyproject.toml` dependency-group placement, not on any doc's claim.
