# U5 Phaser Visual-Parity Task Pack

## Task V1 - Recompose the Menu Dock

### Status

completed

### Goal

Make the Phaser menu read as the same complete mobile-game shell as the accepted Grapes P0.

### Why Now

The current menu has the correct semantic carriers but presents them as a sparse wireframe, so a comparison of the two authoring approaches would be misleading.

### User Lens

The user should immediately recognize a title, hero area, progression path, currency counter, and lower three-button shop/play/settings dock.

### Pre-Shot Targets

- `menu`

### Repro Setup

- Route: immutable Phaser review server
- Viewport: 390 x 844 CSS pixels
- Fixture: accepted P0 candidate
- State: `menu`

### Acceptance Criteria

- Shop, Play, and Settings form one lower dock, with Play visually primary.
- Hero and progression areas have visible composition surfaces.
- All existing menu semantic IDs remain present exactly once.

### Expected Visual Result

A complete light mobile menu with a clear hierarchy and no floating top-right navigation controls.

### Constraints

- All authority changes must be saved and generated by Phaser Editor 5.0.2.
- Visual companions must not carry the Semantic component.

### Out of Scope

- Final game-specific art direction and mechanics.

### Verification

- Compare `screenshots/pre/menu.png` with `screenshots/post/menu.png`.
- Render proof must retain the exact semantic ID set.

### Spawn Rules

- If the visual repair requires changing the cross-lane semantic contract, record a follow-up instead of expanding V1.
- If acceptance is partial, append another journal iteration.

## Task V6 - Prove the Authoring UX

### Status

completed

### Goal

Make Phaser Editor usable for constrained copy and asset selection without opening unsafe authority seams.

### Why Now

Visual parity alone does not prove the Phaser-native approach is a viable design frontend.

### User Lens

The user should be able to find the curated assets and see allowed copy update while typing, with locked commerce/reward facts protected.

### Pre-Shot Targets

- Phaser Editor with the catalog panel open
- Inspector with an allowed copy field selected

### Repro Setup

- Route: loopback-only Phaser Editor over SSH port forwarding
- Viewport: desktop editor
- Fixture: clean accepted P0 scratch
- State: Menu scene and catalog panel

### Acceptance Criteria

- Catalog UI visibly surfaces all 24 entries and required R9 metadata.
- Live preview accepts only proven editable semantic Text carriers.
- Locked facts, nonsemantic text, and metadata fields are excluded.

### Expected Visual Result

A discoverable read-only catalog and a constrained per-keystroke copy workflow.

### Constraints

- Editor remains loopback-only; plugins use no network, storage, or eval APIs.

### Out of Scope

- Drag-and-drop asset replacement automation.

### Verification

- Real-Editor screenshot and automated plugin probes/tests.

### Spawn Rules

- Native custom-pane polish can follow after v1 usability is proven.
